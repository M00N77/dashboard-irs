# IRS Dashboard (Individual Registration System)

[Demo](https://dashboard-irs.vercel.app)

Современная веб-платформа для управления обращениями и учёта данных граждан, разработанная на основе анализа портала ППК РЭО. Проект трансформирует концепцию тяжёлых корпоративных СЭД в быстрый, адаптивный SPA с фокусом на UX и производительность.

## 🚀 Функционал

- **Управление обращениями:** Модульная система обработки заявок с детальной карточкой человека (статусы, источники, категории, сроки).
- **Master-Detail интерфейс:** Интуитивная навигация между списком записей (`/registry`) и детализированной карточкой (`/registry/:id`) с вкладками: общие сведения, семья, образование, жильё, обращения.
- **Интерактивная аналитика:** Дашборд (`/dashboard`) с визуализацией данных (статусы обращений, категории, возрастные группы) через Recharts.
- **Фильтрация и пагинация:** Поиск по ФИО, фильтр по статусу и региону с серверной пагинацией через React Query.
- **Адаптивность:** Полная оптимизация под мобильные устройства (Touch-friendly табы, компактная верстка таблиц, responsive-графики).

## 🏛 От референса к реализации

Проект разработан на основе анализа портала управления обращениями ППК РЭО ([target-tech.ru/portal-ppkreo](https://www.target-tech.ru/portal-ppkreo)). Ключевая задача — не скопировать интерфейс, а **спроектировать современную SPA-систему на той же предметной области**.

### Что взято из референса

- **Модель данных «Обращения» (Appeal):** Введена сущность с категориями, напрямую связанными с деятельностью ППК РЭО — ТКО (твёрдые коммунальные отходы), ЖКХ, жалобы, запросы информации.
- **Статусы обработки:** Новое, в работе, решено, отклонено, перенаправлено — полный цикл обработки заявки.
- **Источники обращений:** Портал, email, телефон, почта, личный приём — мультиканальная модель приёма заявок.
- **Профиль заявителя:** Вместо абстрактного «пользователя» реализован профиль гражданина со связанными обращениями, семьёй, образованием и жильём.

### Что переработано (современный стиль)

- **Material UI (MUI):** Государственные СЭД часто выглядят как перегруженные таблицы из нулевых. Проект использует Material Design — чистую компонентную систему с адаптивной сеткой, бейджами статусов (`Chip`), интерактивными графиками.
- **SPA без перезагрузок:** Асинхронная загрузка данных, лоадеры (`Skeleton`), мгновенная навигация — пользователь не ждёт перезагрузки страниц.
- **Валидация на лету:** Формы с мгновенной обратной связью через React Hook Form + Zod.

### Другая компоновка

- **Картотека как точка входа:** Корневой маршрут (`/`) ведёт в картотеку, а дашборд вынесен в отдельный lazy-маршрут — пользователь сразу видит данные, без загрузки тяжёлых графиков.
- **Master-Detail с вкладками:** Вместо бесконечной формы — логичные табы (Общие сведения, Семья, Образование, Жильё, Обращения). Каждая секция загружается и редактируется независимо.
- **Разделение Summary/Details:** API списка возвращает только поля для таблицы; полный профиль с вложенными связями подтягивается только при открытии карточки — современный архитектурный подход, берегущий трафик.

## 📐 Архитектурные решения

### Feature-Sliced Design (FSD)

Выбран FSD как методология организации кода, доказавшая свою эффективность в средних и крупных React-проектах. Слои (`app`, `pages`, `widgets`, `features`, `entities`, `shared`) формируют строгую иерархию зависимостей — код верхнего слоя не может импортировать из нижнего. Это предотвращает циклические зависимости и упрощает навигацию по проекту.

### MSW для мок-данных

MSW (Mock Service Worker) перехватывает сетевые запросы на уровне Service Worker, а не подменой глобального `fetch`. Преимущества:

- Приложение не знает, что работает с моками — код API-слоя идентичен боевому.
- В панели Network DevTools видятся настоящие HTTP-ответы.
- Не нужно переключать «мок-режим» в коде — достаточно условия `import.meta.env.DEV`.
- Мок-данные генерируются скриптом `npm run generate:mock` с использованием Faker.

### React Query + server-side пагинация

React Query берёт на себя кэширование, фоновую перезагрузку и retry упавших запросов. Пагинация, поиск и фильтрация выполняются на сервере (в MSW-хендлере), а React Query с `placeholderData` (keepPreviousData) делает переключение страниц мгновенным без потери предыдущих данных. Такой подход масштабируется на реальный бэкенд без переписывания клиентского кода.

### Разделение PersonSummary и PersonDetails

Типы разделены намеренно: `PersonSummary` содержит только поля, необходимые для таблицы-списка, а `PersonDetails` — полные данные с вложенными связями (семья, образование, жильё, обращения). Это повторяет практику реальных REST API, где эндпоинт списка не возвращает вложенные объекты ради производительности.

### Типобезопасность форм

Используется `z.input<T>` для синхронизации generic-типа `useForm` с тем типом, который ожидает `zodResolver` (input-тип схемы Zod), исключая ошибки рассинхронизации между схемой валидации и формой.

### Инвалидация кэша дашборда

Мутации обращений и общих данных человека (семья, образование, жильё) не только сбрасывают кэш соответствующей карточки (`['person', id]`), но и инвалидируют запрос `['stats']` дашборда. Это гарантирует, что при переходе на `/dashboard` метрики и графики отображают актуальные данные, несмотря на `staleTime: 5 мин`.

### Обработка пустых ответов API

`apiClient` корректно обрабатывает HTTP-статусы `204 No Content` и `205 Reset Content`, возвращая `null` без попытки парсинга JSON. Это необходимо, поскольку MSW-хендлеры удаления обращений и записей семьи/образования возвращают `204` без тела ответа.

### Метрика активных обращений

Дашборд отображает показатель «Активные обращения» (`activeAppeals`), который вычисляется как **количество обращений** со статусом `new` или `in-progress` (а не количество уникальных граждан). Это обеспечивает точное соответствие между подписью карточки и отображаемым значением.

```bash

git clone https://github.com/M00N77/dashboard-irs.git

npm install
```

## 📈 Производительность и Web Vitals

Проект прошёл оптимизацию по ключевым метрикам Core Web Vitals:

### LCP (Largest Contentful Paint) — < 2.5s

- **Приоритетный маршрут:** Картотека (`/registry`) загружается синхронно, без ожидания lazy-чанка — первый контент появляется сразу после парсинга бандла.
- **Font-display: swap:** Roboto предзагружен через `<link rel="preload">` с `display=swap` — текст виден сразу в системном шрифте, без FOIT.
- **Смещение фокуса LCP:** Статический заголовок "ИРС Дашборд" в AppBar увеличен (`h5` вместо `h6`), а динамическое ФИО в карточке понижено (`h5` вместо `h4`) — LCP-элементом назначается стабильный элемент шапки, а не ждущий API.
- **Отложенная загрузка моков:** Загрузка 7.9MB `persons.json` вынесена после `createRoot` — первый рендер не блокируется.

### CLS (Cumulative Layout Shift) — 0

- **Скелетоны с точными размерами:** `<Skeleton variant="rectangular">` с `height` и `width`, совпадающими с финальным контентом (ФИО `32px`, кнопка назад `40px`, пагинация `36px`).
- **Явные ширины колонок:** У каждой колонки таблицы картотеки прописана `minWidth` (ФИО 25%, Дата рождения 15%, Пол 10%, Статус 15%, Регион 35%) — макет не перестраивается при замене скелетона на текст.
- **Скелетон таблицы:** 10 строк с идентичной структурой шапки и телом — CLS на странице картотеки равен 0.
- **Фиксированные `lineHeight`** для заголовков h1–h4.
- **Responsive-графика:** Использование `<ResponsiveContainer>` из Recharts для предотвращения переполнения и релайаутов на мобильных устройствах.

### FID / INP (Interaction to Next Paint) — < 100ms

- **Неблокирующий main thread:** MSW delay сокращён со 100-800ms до 50-150ms.
- **lazy-загрузка тяжёлых зависимостей:** Дашборд, редактируемые секции карточки и их формы — каждый модуль загружается только при навигации.
- **React Query staleTime (5 минут):** Повторные visits не делают лишних запросов.

## Поведение моков-мутаций

Все мутации (добавление/удаление записей, изменение статуса обращения, обновление
данных человека) работают через MSW-обработчики в памяти браузера. После сохранения
данные обновляются в in-memory кэше, но **не сохраняются на диск** (файл
`public/mock-data/persons.json` не изменяется). При перезагрузке страницы оригинальные
данные из JSON загружаются заново.

Это ожидаемое поведение для мок-окружения. При разработке с реальным API данные
будут сохраняться в БД.
