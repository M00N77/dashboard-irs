### Ключевые технические решения

| Решение | Обоснование |
|---|---|

| MUI | Готовая дизайн-система для дашбордов, таблиц, карточек |

| MSW | Мок-сервер уровня Service Worker — не влияет на код приложения |

## Этап 1 — Мок-данные и API-слой

**Цель:** Сгенерировать offline-датасет (5000 записей), типизировать API-ответы, написать MSW-хендлеры.

### Чеклист

- [ ] Установить `@faker-js/faker`, реализовать offline-генератор с `faker.seed(42)` — 5000 персон
- [ ] Сгенерировать связанные сущности: `family`, `education`, `employment`, `housing`
- [ ] Описать типы `PersonSummary` (для списка) и `PersonDetails` (с вложенными связями) — намеренно разделены
- [ ] Создать `shared/api/client.ts` (инстанс axios/fetch с base URL)
- [ ] Создать `shared/api/persons.api.ts` (функции `getPersons`, `getPersonById`, `updatePerson`)
- [ ] Создать `shared/api/stats.api.ts` (функция `getStats`)
- [ ] Написать MSW-хендлеры: `GET /persons` (пагинация, поиск, фильтр, сортировка на сервере)
- [ ] Написать MSW-хендлер: `GET /persons/:id`
- [ ] Написать MSW-хендлер: `GET /stats` (агрегаты считаются в хендлере, не на фронте)
- [ ] Добавить искусственные задержки (100–800ms) и ~5% случайных ошибок (404/500)

### Ключевые технические решения

| Решение | Обоснование |
|---|---|
| `faker.seed(42)` | Детерминированная генерация — консистентные данные при каждом запуске |
| Разделение `PersonSummary` / `PersonDetails` | Как в реальных REST API — список не тащит вложенные связи |
| Серверная пагинация, поиск, фильтр, сортировка | Имитация реального бэкенда, валидация подхода до написания UI |
| Агрегаты в MSW-хендлере | Фронт не делает вычислений — только отображает |
| Случайные ошибки + задержки | Проверка обработки ошибок, лоадеров и retry на раннем этапе |

### Критерий «готово»

MSW запускается, `GET /persons?page=1&pageSize=20&search=…` возвращает корректные данные, `GET /persons/:id` — карточку, `GET /stats` — агрегаты. В панели Network DevTools видны ответы мок-сервера.

## Этап 2 — Картотека (список)

**Цель:** Построить страницу списка граждан с серверной пагинацией, поиском, фильтрацией и сортировкой.

### Чеклист

- [ ] Установить `@tanstack/react-table` (headless)
- [ ] Создать страницу `pages/registry` с таблицей
- [ ] Реализовать серверную пагинацию через React Query (`useQuery` с page/pageSize)
- [ ] Подключить TanStack Table + MUI-компоненты для рендеринга ячеек/заголовков
- [ ] Реализовать фильтры (текстовый поиск, выпадающие списки по полу/статусу и т.д.)
- [ ] Состояние поиска/фильтра/пагинации/сортировки — в URL (`useSearchParams`)
- [ ] Вывести счётчик общего количества записей
- [ ] Клик по строке → переход на `/persons/:id`
- [ ] Skeleton/loader на время загрузки

### Ключевые технические решения

| Решение | Обоснование |
|---|---|
| TanStack Table (headless) | Полный контроль над рендерингом + MUI-стилизация; не тянет тяжёлый DataGrid |
| `useSearchParams` | Состояние списка в URL — шаринг, возврат к результатам поиска после редактирования |
| Серверная пагинация через React Query | `keepPreviousData`, фоновая перезагрузка, отмена устаревших запросов |

### Критерий «готово»

Список отображается, пагинация работает, поиск и фильтры обновляют URL и данные, строка ведёт в карточку.

- [ ] Мутация `updatePerson` через `useMutation` + инвалидация кэша списка и карточки

### Ключевые технические решения

| Решение | Обоснование |
|---|---|
| React Hook Form + Zod per tab | Валидация на уровне каждого таба; общая схема — на уровне формы |
| `useMutation` + query invalidation | Мутация автоматически сбрасывает кэш списка — консистентность без ручного refetch |
| Режим чтения/редактирования | Пользователь не может случайно изменить данные; явное действие «редактировать» |

### Критерий «готово»
